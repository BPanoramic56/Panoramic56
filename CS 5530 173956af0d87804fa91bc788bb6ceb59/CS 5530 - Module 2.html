  • ER Model

    Entity → A concrete “object” distinguishable from other entities, described by a set of attributes

    • uID (string), name (string), DOB (date)

    Entity Set → Set of all entities of the same type

    • All students, all cars, all library books
    • All entities in the set have the same attributes
    • An entity set has a key attribute

    Relationship → Relationship between 2 or more entities

    • Can have attributes as well

      Ex: “Danny works in SoC since 2010” → ‘since 2010’ is an attribute of the relationship, it does not belong to Danny or SoC

    Relationship Set → The set of relationships between entity sets

  • Intro to ER Diagrams
    • Entity Set

      A square represents the entity set, circles represent attributes of the set and underlines represent parts of the key

    • Relationship Set

      Diamonds represent the relationship set and they do not need a key. Circles connected to the diamond represent that relationship’s attributes

      Inter-Relationships: Relationships can connect entity’s set to itself (inter-relationship). These need to be labeled

      Cardinality: Determine constraints of the relationship

      • Many-to-Many: Entities in sets one and two can participate in multiple relationships
      • One-to-Many: Entities of set one can participate in multiple relationships, but entities of set two can only participate in one
      • One-to-One: Entities of sets one and two can only participate in one relationship

      Participation Constraint: Double lines indicates all entities in the set must participate in the relationship

  • Advanced ER patters

    Hierarchical Types

    • Triangles represent inheritance
    • Entity or Attribute?

      Is it a complex data that needs its own keys or individuality → Entity

      Can an entity have more than one of them? → Entity

      Does the data make sense in another relationship → Entity

      Just some examples, there are several ways of designing the same database

  • Ternary Relationships

    A diamond may be connected with three (or more) entities

    Example: An employee can live in an address during a certain duration, or, an address can have an employee living in it during a certain duration

  • Weak Entities

    Represented by a double rectangle

    A double diamond is used to represent the supporting relationship set. It must have a cardinality of 1 on the strong side and a weak entity must participate

    Weak entities don’t have keys, they have partial keys, represented by a dashed underline

    They can have normal relationships too

    Something that cannot be identified by its own attributes

    It is identified by a combination of its own attributes and another entity’s key

    Example: Course vs Class

    Course (Strong Entity Set)

    • Subject
    • Number
    • Name
    • Description

    Class (Weak Entity Set)

    • Semester
    • <Reference to course>

    Weak entity sets can be weak themselves, as long as there is a strong entity at the end, and their keys compound together

  • ER → RM Algorithm

    Basic Algorithm:

    • 1. Every entity set becomes a schema
      1. Attributes become columns
        1. They are set as NN (Not Null)
      1. Set the key attribute(s) as the primary key
    • 2. Relationship sets might become schemas depending on cardinality, otherwise they are merged
      • Many-to-Many: becomes it’s own table → Combination of the two keys of the relationship and that becomes the new table’s key)
      • One-to-Many: Merge the relationships into the M-side entity

        A foreign key (FK) is used to denote the relationship’s destination, and it can be Null

        If participation is required, the attribute is set as NN, otherwise, Null is allowed

      • One-to-One: Merge relationship into one of the other tables

        Same as One-to-Many, but the foreign key now has to be unique

      • Supporting relationships

        Treat as One-to-Man, merge into M side

        Same procedure, except supporting key is combined with partial key

  • Double Participation

    If participation is required on both sides of a relationship (can happen in other participation/cardinality restraints too), that becomes difficult to capture with schema design

    • How can you add attributes as foreign keys if those entities don’t even exist yet? (Chicken-egg problem)
    • Instead, enforce with SQL commands

    Incompatible Constraints

    If some combination of participation/cardinality can not be represented in tables, cardinality takes precedence, and enforce participation through software (or make software aware that it is not enforced

  • Hierarchy to Tables
    • Method 1

      Create a schema for every entity set

      Copy the primary key for the base type as a primary key for the derived type → is-a relationship

      The other attributes will be referenced by the existence of the foreign key, so they do not need to be copied to the derived types

    • Method 2

      No schema for the base type, only derived types

      All attributes from base type are copied to derived type as primary keys

    • Which one to use

      Method 1:

      • If an entity can be two different derived types
      • If the base type has relationships
      • E.g.: Attributes, such as SSN, need to have a direct relation to name, regardless of which table they appear

      Method 2:

      • If the base type is “abstract”. The base type can never be instantiated
      • E.g.: An employee has to either be a salaried employee or an hourly employee, it cannot be neither
