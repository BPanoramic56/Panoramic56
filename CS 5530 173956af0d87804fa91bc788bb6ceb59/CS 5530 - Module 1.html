Relational Databases

  • Structured Data Storage
    • Related data are grouped together or stored next to each others in relations (tables)
    • Non directly related information are stored in separate relations (separate tables)
    • Cells do not need to be unique, but rows are - there is some unique cell in all rows
    • Key → A unique identifier for each row. All relations must have this, if no field is specifically unique, an ID number for the rows has to be created
    • Relations can have entities or relation-based entities
      • One relation between name → phone number, one for name → card number, one for card number → book, etc
  • Nomenclature

    Attribute → A name and a type (column heading)

    Schema → Combination of all attributes + Relation name

    Instance → The values in a table (a set of tuples)

    Tuples → One row of the data (one value for each attribute)

    Relation → The Schema + Instance

    Temporary Relation → Relations are not necessarily a fixed part of the database (a filter will only show relevant tuples, that is a temporary relation)

  • Row Uniqueness

    Key: Unique identifier of each row

    • A set of attributes is a key if: it is a superkey and no proper subset of its attributes is a superkey (the empty set is not usually considered a superkey)
    • Keys define what data is valid (not vice-versa)

    Superkey: A set of attributes is a superkey if no two rows are allowed to have the same values in those columns (no duplicates)

    • Ex: A1 and A2A_1 \space and \space A_2A1​ and A2​﻿ are superkeys

      No items of A1A_1A1​﻿ can repeat an no items of A2A_2A2​﻿ can repeat

    • Ex: A1A2 (compound)A_1A_2 \space(\text{compound})A1​A2​ (compound)﻿ is a superkey

      Items of A1A_1A1​﻿ can repeat, items of A2A_2A2​﻿ can repeat, but A1A2A_1A_2A1​A2​﻿ items cannot

    We define the superkeys, not the data

    Critical for DBMS underlying operation

    Keys can be composed of more than one column

  • Multi-key Table
    • In multi-key relations, one of the keys need to be a primary key, specified by the Database Administrator (DBA), the other keys are called candidate keys
      PRIMARY KEY (sID, cID)
UNIQUE (sID, cID)
    • There is no difference between primary and candidate keys in terms of uniqueness constraints (both will be enforced equally)
      • There is, however, performance differences

        The primary key should be small in terms of data size (easy copying) and ideally an integer type (fast comparison)

    • Examples
      • {sID,cID}\{sID, cID\}{sID,cID}﻿ compounded table
        • A student cannot enroll in the sane course more than once)
      • What if we made it two separate keys? {sID} and {cID}\{sID\} \space and \space \{cID\}{sID} and {cID}﻿
        • A course can only have one student and each student can only have one course
      • What if a student can not take the same course twice and can not receive the same grade twice (in any course)?
        • We need two keys:

          {sID,cID}\{sID, cID\}{sID,cID}﻿ → student can not take the same course twice

          {sID,Grade}\{sID, Grade\}{sID,Grade}﻿ → student can not receive the same grade twice (in any course)

      • A student can not take the same course twice, and no students can receive the same grade in the same course

        Candidate Key: {cID,Grade},Primary Key: {sID,cID}\text{Candidate Key: }\{cID, Grade\}, \\ \text{Primary Key: }\{sID, cID\}Candidate Key: {cID,Grade},Primary Key: {sID,cID}﻿

      • A student can only take one course, and no two students can earn the same grade (among all courses)

        Candidate Key: {Grade},Primary Key: {sID}\text{Candidate Key: }\{Grade\}, \\ \text{Primary Key: }\{sID\}Candidate Key: {Grade},Primary Key: {sID}﻿

  • Foreign Key

    Attribute whose values are a key in another table (like a pointer in C++)

    • Foreign Keys are not necessarily unique itself
    • Referential Integrity
      • References between values should always be meaningful
      • This is an invariant - gold true at all times
      • SQL supports this trough Foreign Keys
    • If a column is specified as a foreign key:
      • We can not insert values that don’t exist in the other table
      • When a value is removed from the other table, it is either disallowed of the foreign key is updated
    • References have directions

      The initial table can have keys that are not present as foreign keys in the referencing table, but all keys in the referencing table need to have an equivalent key in the initial one

    • Ways of enforcing integrity
      • Cascade

        If row of initial table is deleted, the equivalent foreign key rows are also deleted

      • Set Null

        If row of initial table is deleted, nullify the foreign keys (set them to NULL), but keep the rows the same otherwise

      • Restrict

        Disallow changes to referenced table (deleting the rows in the original table that have equivalent foreign keys is not allowed

        Makes sense if there is some action that needs to happen in case a row is deleted
